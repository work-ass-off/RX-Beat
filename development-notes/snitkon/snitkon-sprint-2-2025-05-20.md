# Sprint 2:  Routing & Signals — 2025-05-20

- **What was done:** Expanded the existing GitHub Actions pipeline to support fully automated continuous delivery (CD) to Vercel. Configured global repository environment variables (`VERCEL_TOKEN`, `VERCEL_ORG_ID`, `VERCEL_PROJECT_ID`) directly within the `jobs.env` block to keep deployment commands clean and secure. Implemented a highly resilient three-stage deployment architecture:
  1. **Sprint Preview:** Generates isolated preview links for internal feature testing when creating Pull Requests into `sprint-*` branches.
  2. **Pre-merge Production:** Triggers a dry-run production build check when a PR is opened into the `develop` branch to prevent any broken code from reaching the live environment.
  3. **Production Release:** Automatically deploys verified changes to the live, user-facing website immediately after a Pull Request is successfully merged (via a `push` event) into the `develop` branch.

- **Problems:** - Ran into a deployment failure when using the `--prebuilt` flag because Angular builds the project into its default `dist/` directory, while Vercel's prebuilt flag expected output specifically inside the `.vercel/output` folder.
  - Encountered a race condition and duplicate deployment "noise" in GitHub Pull Requests because the native `Vercel Bot` integration was firing automatically on every Git push, completely bypassing the strict linting and formatting gates established in GitHub Actions.

- **Solutions:** - Dropped the `--prebuilt` flag from the workflow steps and shifted to standard `npx vercel deploy` and `npx vercel deploy --prod` commands, allowing Vercel to cleanly pick up and upload the verified production assets generated by `npm run build`.
  - Hardened the pipeline by configuring the **Ignored Build Step** inside the Vercel dashboard settings with a custom `exit 0` override script. This effectively silenced the automated Vercel Bot, giving GitHub Actions exclusive authority over when a deployment occurs.
  - Excluded the `main` branch from the heavy application CI/CD trigger rules since it is strictly reserved for standalone Markdown documentation (`diary`), preventing unnecessary build-minute consumption.

- **What I learned:** - Deeply understood the behavioral distinction between Git-level operations and GitHub Action context events, discovering that a Pull Request Merge is technically processed as a `push` event on the target branch.
  - Learned how to manage Vercel CLI deployments securely via environment tokens without exposing keys in the workflow output logs.
  - Mastered the concept of *Production Previews*, which allow a team to safely review and QA a web application under real-world production configurations before committing the changes to the live domain.

- **Plans:** Continue improving my application deployment skills and deeply exploring modern DevOps and CI/CD tools to further optimize development workflows.
